# 面试讲述脚本 · STAR 法则

> 本文档提供三个时长版本的讲述脚本，根据面试节奏选用。核心永远是**多 Agent 协作方法论**与**Harness 量化评测体系**。

---

## 一、3 分钟版（电梯演讲 / 自我介绍后追问）

### S - Situation（背景，20 秒）

我之前在准备大数据开发岗面试时，发现网上面经有三个痛点：**题目杂乱无章、答案错误过时、刷题重复率高**。市面上的题库要么是通用八股、要么是人工整理、质量参差不齐，而且没有量化标准判断"这个题库到底好不好"。

### T - Task（任务，15 秒）

所以我想做一个**全自动、可量化、岗位定向**的大数据面试题库——不是简单让 AI 出题，而是搭建一条多 Agent 协作流水线，配一套量化评测引擎，让题库质量可度量、可迭代。

### A - Action（行动，90 秒）

我设计了**五阶段流水线 + 五个专职 Agent**：

第一阶段 collector 定向爬取牛客网大数据面经，过滤无关岗位；第二阶段 cleaner 做分类、去口语化、设计选项、统一编号；第三阶段 generator 基于知识图谱生成变式题、派生代码题、标注四级难度；第四阶段两个阅卷 Agent **并行**补全简答和代码标准答案；最后第五阶段我自己写了套 **Harness 量化评测引擎**，从重复率、答案准确性、考点覆盖率、难度均衡度四个维度自动打分。

关键技术决策有三个：一是 Agent 之间**不直接通信**，所有信息经主理人中转，避免上下文污染；二是用 **Contents API 逐文件推送**绕过 github.com 网络阻断；三是评测引擎用**纯 Python 零依赖**实现，Jaccard 相似度做重复检测，参数密度做答案质量量化。

### R - Result（结果，35 秒）

最终产出 **112 道题 + 74 道标准答案**，覆盖 9 大技术栈。Harness 首次评测发现 L1 基础题严重不足——只有 3 道，偏差 -12%，触发修复闭环，我补了 12 道题后重新评测，**四维全部 PASS**。答案参数密度达到平均每题 7.4 个真实配置参数，对比表格 260 行，代码复杂度标注覆盖率 204%。全程 5 个 Agent 零人工干预，54 分钟跑完全流水线。

## 二、5 分钟版（项目深聊 / 面试官说"详细讲讲这个项目"）

### S - Situation（45 秒）

准备大数据面试时，我系统调研了牛客网等平台的面经生态，发现三个结构性问题：

第一，**题目杂**——牛客网是全岗位混在一起，前端后端测试大数据全有，找大数据题像大海捞针；第二，**答案差**——网友回答口语化、概念有误、参数过时，比如有人还在说 Kafka 用 high-level consumer，这早就 deprecated 了；第三，**无标准**——没人能说清楚"一个好题库应该长什么样"，重复率多少算高？难度怎么分布合理？全是主观感受。

### T - Task（30 秒）

我的目标不是做一个题库，而是做一套**可复现、可量化、可迭代**的题库生产体系。核心挑战是：怎么让 AI 出题质量可控？怎么证明题库"好"而不是"我觉得好"？

### A - Action（180 秒）

我搭了一条**五阶段多 Agent 协作流水线**，每个阶段一个专职 Agent，串行联动、阶段内并行：

**Phase 1 采集**：collector Agent 读牛客网面经报告 + WebSearch 补充，定向抓大数据岗，精准过滤 Java 八股和前端测试，产出 40 道原始题。

**Phase 2 标准化**：cleaner Agent 做四件事——题型分类（选择/简答/代码）、去口语化（"说一下"改标准问句）、残缺剔除、编号体系建立。

**Phase 3 扩展**：generator Agent 做变式出题——场景变式、角度变式、深度变式、反向变式四种类型；再基于简答题考点派生 10 道手撕代码题；最后标注四级难度和知识图谱关联。从 40 道扩到 100 道。

**Phase 4 并行阅卷**：saq-grader 和 code-grader 两个 Agent 同时工作，分别补 50 道简答标准答案和 20 道代码参考实现。简答题按"原理+流程+参数+对比表+追问+生产踩坑"六件套输出，代码题按"可运行代码+中文注释+复杂度+测试用例+方案对比"五件套输出。

**Phase 5 量化质检**：这是我自己写的 Harness 引擎，四个维度——题目重复率用 Jaccard 相似度全量比对、答案准确性用真实参数密度量化、考点覆盖率检测 9 大技术栈缺失、难度均衡度算 L1-L4 占比偏差。

**协作机制**上，Agent 之间不直接通信，所有信息经主理人中转，这样避免上下文污染、也便于追溯。遇到 github.com 网络不通，我用 Contents API 逐文件推送绕过，还触发过 GitHub Secret Scanning 拦截硬编码 token，最后改成环境变量读取。

### R - Result（45 秒）

最终 **112 道题 + 74 道答案**，9 大技术栈全覆盖。Harness 闭环价值最典型的体现是：首次评测 L1 只有 3 道、偏差 -12% 直接 FAIL，我补 12 道后重测偏差收敛到 -1.6% PASS。答案量化指标：参数密度 7.4/题、对比表格 260 行、复杂度覆盖 204%、重复率 0。整套流水线 54 分钟跑完，5 个 Agent 零人工干预。代码和 Agent 定义都开源在 GitHub 了。

## 三、10 分钟版（终面 / 技术答辩 / 专门讲项目的时间）

> 在 5 分钟版基础上，每个阶段补充**技术细节 + 设计权衡 + 踩坑经历**。

### 补充的技术深挖点（按面试官可能追问的方向准备）

**1. 为什么用多 Agent 而不是一个模型一把梭？**

单模型出题有三个问题：上下文窗口有限、角色混乱、质量不可控。多 Agent 的核心价值是**单一职责**——collector 只管采集、cleaner 只管清洗，每个 Agent 的 prompt 聚焦一件事，质量天花板更高。代价是协调成本，但我用主理人中转模式解决了。

**2. Harness 评测怎么保证不是"自己考自己"？**

这是个好问题。目前确实是 Agent 出题 + 脚本评测，没有独立基准库。我的做法是：评测维度选**可客观计算的指标**——Jaccard 相似度是数学运算、参数密度是正则匹配、占比是统计，不依赖主观判断。下一步可以引入外部基准知识库做交叉校验。

**3. 难度分级怎么保证客观？**

我定了明确标准：L1 是概念记忆（"Kafka 是什么"）、L2 是原理流程（"HDFS 读写流程"）、L3 是方案设计（"数据倾斜怎么解决"）、L4 是源码级架构权衡（"Unaligned Checkpoint 原理"）。Generator Agent 按这个标准标注，Harness 再统计占比校验。首次 L1 偏少就是因为 generator 倾向出难题，Harness 发现后触发补全。

**4. 变式出题怎么避免"换汤不换药"？**

四种变式类型有明确差异：场景变式换业务背景、角度变式换考查视角、深度变式往底层挖、反向变式把正向问题反过来问。Harness 的重复率检测是兜底——Jaccard 相似度 >= 0.5 就标重复，实测 112 道题 6216 对比对，0 重复。

**5. 最大的技术挑战是什么？**

两个。一是 github.com 网络不通，git push 直接 SSL 握手失败，我排查发现 api.github.com 可达但 github.com 不行，最后用 Contents API 逐文件推送绕过，还踩了 Secret Scanning 的坑。二是 Harness 闭环——第一次跑发现 L1 严重不足，这不是写个脚本就完事，要设计"检测→FAIL→修复→重测→PASS"的完整闭环，还要把修复过程记录在报告里。

## 讲述要点提醒

| 顺序 | 必须提到 | 时间占比 |
|------|---------|---------|
| 1 | 痛点：题目杂/答案差/无标准 | 15% |
| 2 | 多 Agent 流水线（五阶段五角色） | 30% |
| 3 | Harness 四维量化评测（核心亮点） | 25% |
| 4 | 闭环修复案例（L1 从 3→15） | 15% |
| 5 | 量化结果（112题/74答案/ALL PASS） | 10% |
| 6 | 开源 GitHub | 5% |

> **铁律**：永远先讲"为什么这么做"，再讲"做了什么"。面试官关心的是你的思考过程，不是流水账。
