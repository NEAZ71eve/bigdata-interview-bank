# 多 Agent 对抗协作架构（v2 升级版）

> v1 是"单向分工"——五个 Agent 各干各的，换成一个模型分五次调用效果一样。
> v2 是"对抗+交叉+闭环"——不同角色有不同目标，通过博弈提升质量，这才是多 Agent 不可替代的价值。

---

## 一、核心问题诊断（v1 为什么不行）

| v1 的问题 | 具体表现 | 后果 |
|---------|---------|------|
| 没有对抗 | generator 出题无人挑毛病，grader 写答案无人审查 | 质量靠单个 Agent 一次性产出，无迭代 |
| 没有交叉 | saq-grader 和 code-grader 各干各的，不互校 | 同一考点可能矛盾、参数可能不一致 |
| 质检是脚本不是 Agent | Harness 只算指标，不能判断"哪道题需重写"、不能触发重做 | 闭环靠人工补题，不是自动的 |
| 单向流水线 | A→B→C→D→E 一路向下，没有回路 | 错误会一路传到底，中途无纠偏 |

**一句话：v1 是"分工"不是"协作"，是"流水线"不是"博弈"。**

---

## 二、v2 架构：对抗 + 交叉 + 闭环

### 2.1 新增三个 Agent 角色

| 新角色 | 目标 | 与谁对抗 | 价值 |
|--------|------|---------|------|
| **reviewer** | 挑毛病（找错误/歧义/过时/标错难度） | vs generator（出题方） | 出题质量经多轮博弈提升 |
| **cross-reviewer** | 交叉校验两个 grader 的答案一致性 | vs saq-grader + code-grader | 消除跨题型矛盾 |
| **qa-agent** | 智能判断哪道题需重写 + 触发对应 grader 重做 | vs 全体产出方 | 质检闭环自动化 |

### 2.2 新流水线（含对抗回路）

```
Phase 1-3: 采集 → 标准化 → 扩展出题（同 v1）
                    ↓
Phase 4: ★对抗迭代★
  generator 出题 → reviewer 审查 → generator 修改 → reviewer 再审
  （多轮直到 reviewer 认可或达到上限轮次）
                    ↓
Phase 5: 双 grader 并行阅卷（同 v1）
                    ↓
Phase 6: ★交叉审查★
  cross-reviewer 校验 saq 答案与 code 答案的一致性
  （参数是否一致、知识图谱关联是否成立、同一考点说法是否矛盾）
                    ↓
Phase 7: ★智能质检闭环★
  qa-agent 不只算指标，还：
  1. 判断哪些题的答案质量不达标（参数密度低/复杂度缺失/审查未通过）
  2. 给出具体修改建议
  3. 自动触发对应 grader 重做该题
  4. 重做后再审，直到达标
                    ↓
交付：ALL PASS
```

### 2.3 对抗机制详解（核心创新）

**generator vs reviewer 的博弈**：

| 角色 | 目标 | 行为 |
|------|------|------|
| generator | 出好题、出全题 | 倾向多出题、出难题 |
| reviewer | 挑毛病、卡质量 | 找歧义、找错误、找标错难度、找过时内容 |

**博弈规则**：
1. generator 产出后，reviewer 逐题审查，输出问题清单（P0/P1/P2 分级）
2. generator 收到清单后**必须修复所有 P0**，P1 选择性修复
3. 修复后 reviewer 再审，只查上轮问题是否解决
4. 最多 3 轮，3 轮后仍有 P0 则标记为"人工介入"

**为什么这比单模型好**：
- 单模型自己出题自己审 = 自己考自己，必然放水
- 两个 Agent 目标不同，reviewer 的 prompt 明确要求"严苛挑毛病"，会真实发现 generator 遗漏的问题
- 多轮迭代让质量收敛，不是一次性赌运气

### 2.4 交叉审查机制

**cross-reviewer 做什么**：
1. **参数一致性**：saq 答案说 Kafka `min.insync.replicas=2`，code 答案里是否一致使用？
2. **知识图谱关联**：generator 标注"SAQ-001 关联 CODE-006"，实际内容是否真的相关？
3. **说法一致性**：同一考点（如 Spark Shuffle）在简答题和代码题里的描述是否矛盾？
4. **难度一致性**：同一考点的简答题和代码题难度标签是否协调？

### 2.5 智能质检闭环

**qa-agent vs Harness 脚本的区别**：

| 维度 | Harness 脚本（v1） | qa-agent（v2） |
|------|-------------------|---------------|
| 能力 | 算指标、输出 PASS/FAIL | 算指标 + **判断哪道题有问题** + **给修改建议** + **触发重写** |
| 闭环 | 人工看报告后手动补题 | 自动触发对应 grader 重做该题 |
| 智能 | 无（纯统计） | 有（理解答案内容，判断质量） |
| 输出 | 报告 | 报告 + 重写指令 + 重做后的答案 |

## 四、v1 vs v2 对比

| 维度 | v1 单向分工 | v2 对抗协作 |
|------|-----------|-----------|
| Agent 数量 | 5 | 8（+reviewer/cross-reviewer/qa-agent） |
| 出题质量保障 | generator 一次性产出 | generator vs reviewer 多轮博弈 |
| 答案一致性 | 无保障 | cross-reviewer 交叉校验 |
| 质检闭环 | 脚本算指标，人工补题 | qa-agent 智能判断+自动触发重写 |
| 错误纠偏 | 一路传到底 | 多个回路纠偏 |
| 多 Agent 不可替代性 | 弱（换单模型一样） | 强（对抗博弈必须多角色） |
