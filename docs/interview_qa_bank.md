# 面试问答预案 · 24 题标准回答

> 按面试官追问路径分 6 类，每题给**回答要点**而非完整话术，避免背诵感。

---

## 一、项目动机与背景（4 题）

### Q1：为什么做这个项目？不是已经有牛客网 LeetCode 了吗？

**要点**：
- 痛点三连：牛客是全岗位混杂（找大数据题像大海捞针）、网友答案错误过时（如 Kafka high-level consumer 早 deprecated）、题库质量无量化标准（重复率多少算高？难度怎么分布？没人说得清）
- 我不是要再做一个题库，而是要解决"题库质量怎么度量"这个问题——Harness 评测引擎才是核心

### Q2：这个项目解决的核心问题是什么？

**要点**：
- 表面：大数据岗题库自动化生产
- 本质：**让 AI 生成内容的质量可量化、可迭代**——这套多 Agent + Harness 的方法论可以迁移到任何内容生产场景（文档生成、代码审查、测试用例等）

### Q3：市面上的 AI 出题工具很多，你这个有什么不同？

**要点**：
- 大多数是"一个模型一把梭"，我的是多 Agent 职责拆分
- 大多数没有质检，我有四维量化评测 + 闭环修复
- 大多数不可复现，我的 Agent 定义和评测引擎都开源

### Q4：这个项目的商业价值在哪？

**要点**：
- 直接价值：大数据求职者刷题、面试官出题
- 方法论价值：多 Agent 流水线 + 量化质检的范式可迁移到企业内容生产（如技术文档自动化、培训材料生成、代码审查流水线）

## 二、架构设计（5 题）

### Q5：为什么用多 Agent 而不是一个模型一次生成？

**要点**：
- 单模型三个问题：上下文窗口有限（100 道题+答案放不下）、角色混乱（又采集又出题又阅卷会互相干扰）、质量不可控（没有分工就没有专精）
- 多 Agent 核心价值：单一职责，每个 Agent prompt 聚焦一件事，质量天花板更高
- 代价是协调成本，用主理人中转模式解决

### Q6：五个 Agent 是怎么协作的？能不能详细讲讲？

**要点**：
- 串行联动：Phase 1 采集 → Phase 2 标准化 → Phase 3 扩展 → Phase 4 并行阅卷 → Phase 5 质检
- Phase 4 内部并行：saq-grader 和 code-grader 同时工作（无依赖）
- 消息中转：Agent 之间不直接通信，所有信息经主理人，避免上下文污染 + 便于追溯

### Q7：为什么不让 Agent 之间直接通信？主理人中转不是效率低吗？

**要点**：
- 直接通信的问题：N 个 Agent 两两通信是 O(N²) 复杂度，上下文互相污染，难追溯
- 主理人中转：O(N) 复杂度，主理人掌握全局视图，可以做质量核验和阶段门禁
- 效率代价可接受——每个阶段也就多一次主理人转发，但换来可控性和可追溯性

### Q8：Agent 的 prompt 是怎么设计的？有什么讲究？

**要点**：
- 结构统一：基本信息表 → 角色指令 → 产出格式 → 重要约束
- 关键设计：明确输入（读哪个文件）、明确输出（写哪个文件、什么格式）、明确质量标准（如简答题"六件套"）、明确回传机制（SendMessage 给 team-lead）
- 示例：cleaner 的 prompt 把四大职责拆得很细（分类/去口语/剔除/编号），每项有具体规则，不是模糊指令

### Q9：如果某个 Agent 产出质量差怎么办？有没有 retry 机制？

**要点**：
- 当前机制：主理人每阶段做核验（文件大小、行数、题目数、标签一致性），不合格不下发下一阶段
- Harness 质检是兜底——L1 不足就是 Harness 发现的，触发主理人补题
- 改进方向：可以给每个 Agent 加自检步骤（产出后自己先跑一遍校验），或引入验证 Agent 做交叉审查

## 三、Harness 评测体系（5 题）

### Q10：Harness 四个维度是怎么选的？为什么是这四个？

**要点**：
- 重复率：题库最基本要求，重复 = 刷题无效
- 答案准确性：答案错了比没有答案更害人
- 覆盖率：技术栈有缺失 = 知识有盲区
- 均衡度：难度断层 = 用户体验差（全是难题新人没法用，全是简单题老手没价值）
- 这四个覆盖了"题目本身质量"和"题库整体质量"两个层面

### Q11：重复率用 Jaccard 相似度靠谱吗？语义相似但字面不同怎么办？

**要点**：
- Jaccard 是 baseline，优点是零依赖、可解释、速度快（6216 对比对秒级完成）
- 局限：语义相似但字面不同会漏检（如"Kafka 怎么保证不丢消息" vs "Kafka 消息可靠性机制"）
- 改进方向：可以加 embedding 余弦相似度做语义层，但需要模型依赖，权衡后当前用 Jaccard 够用（实测 0 重复）

### Q12：答案准确性怎么量化？"准确性"不是主观的吗？

**要点**：
- 我不评"答案对不对"（那需要基准知识库），我评"答案有没有真实参数支撑"
- 逻辑：一道讲 Kafka 可靠性的答案如果没提到 acks=all、min.insync.replicas，大概率不专业
- 做法：正则匹配 20+ 真实配置参数/API，算密度（命中次数/题数），阈值 2/题，实测 7.4/题
- 这是代理指标（proxy metric），不完美但可量化、可复现

### Q13：难度均衡度的 ±7% 容差是怎么定的？

**要点**：
- 经验值：L1 目标 15%，±7% 意味着 8%-22% 都算合格，这个区间既不至于断层（<8%）、也不至于泛滥（>22%）
- 首次 L1 只有 3%（偏差 -12%）直接 FAIL，说明阈值有效——太严会频繁误报，太松失去意义
- 可以按场景调：刷题产品可以放宽 L1，面试冲刺产品可以收紧 L4

### Q14：Harness 怎么接入 CI/CD？

**要点**：
- 引擎退出码 0=ALL PASS / 1=存在 FAIL，可直接接入 CI pipeline
- 每次 Agent 产出后自动跑 Harness，FAIL 则阻断流水线 + 输出修复清单
- 报告输出 harness_report.md，可归档做版本对比（如 v1.0 → v1.1 难度分布变化）

## 四、技术难点与踩坑（4 题）

### Q15：项目最大的技术挑战是什么？

**要点**（讲两个）：
- **网络阻断**：github.com SSL 握手被重置，git push 完全不可用。排查发现 api.github.com 可达（HTTP 200）但 github.com 不行（连接重置）。解决方案：用 Contents API 逐文件推送，绕过 git 协议
- **闭环设计**：Harness 不是写个统计脚本就完事，要设计"检测→FAIL→修复→重测→PASS"的完整闭环。L1 从 3 道→15 道就是这个闭环的真实案例，修复前后对比要记录在报告里

### Q16：GitHub Secret Scanning 拦截是怎么回事？

**要点**：
- 第一个版本的上传脚本里硬编码了 token，GitHub 的 Secret Scanning 自动检测到并拦截（409 Repository rule violations）
- 说明 GitHub 安全机制有效，也提醒我不能把密钥放代码里
- 修复：改成从 GITHUB_TOKEN 环境变量读取，重新上传成功

### Q17：Contents API 逐文件推送有什么局限？

**要点**：
- 每个文件一次 API 调用，11 个文件 11 次 commit，不像 git push 是原子提交
- 大文件（如 125KB 的 phase4_saq_answers.md）要 base64 编码后放 JSON body，有大小限制（GitHub 限制 100MB，但建议 < 1MB）
- 更新文件需要先 GET 获取 sha 再 PUT，多一次往返
- 适合小项目，大项目还是得解决 git push 网络问题

### Q18：多 Agent 协作中遇到过什么坑？

**要点**：
- Generator 倾向出难题——首次 L1 只有 3 道（3%），因为模型天然倾向"有深度"的内容
- 这个问题如果不是 Harness 检测出来，人工根本发现不了——100 道题逐个看难度标签太低效
- 启示：AI 生成内容必须有量化质检，不能靠人眼

## 五、量化结果与数据（3 题）

### Q19：112 道题 + 74 道答案，这个量级有意义吗？

**要点**：
- 量级不是重点，方法论才是——112 道题足够验证流水线可行性
- 真正的价值在量化指标：重复率 0、参数密度 7.4/题、四维 ALL PASS，这些是可度量的质量证据
- 可扩展：加更多 Agent 并行、加更多数据源，题量可以线性增长

### Q20：54 分钟跑完，这个效率怎么样？

**要点**：
- 串行阶段（采集→标准化→扩展）各约 7 分钟，并行阅卷约 12 分钟，质检 1 分钟，修复 8 分钟
- 瓶颈在 LLM 推理速度，不是架构问题
- 如果要优化：Phase 1-3 可以部分并行（cleaner 边收边洗）、阅卷可以分片并行（50 道简答拆给多个 saq-grader）

### Q21：参数密度 7.4/题这个数字怎么来的？可信吗？

**要点**：
- 54 道简答题里正则匹配到 398 次真实参数（acks=all、checkpoint、watermark 等），398/54=7.4
- 正则模式列表在 harness_engine.py 里公开，可复现
- 局限：参数多 ≠ 答案对（可能参数对但逻辑错），但参数少 ≈ 答案不专业，所以作为下限指标有效

## 六、扩展性与反思（3 题）

### Q22：如果要做成产品，还需要什么？

**要点**：
- **用户侧**：Web 界面（刷题/组卷/进度追踪）、Anki 卡片导出、间隔重复算法
- **内容侧**：更多岗位（Java/前端/算法）、更多技术栈、定期更新机制
- **质检侧**：引入外部基准知识库做答案交叉校验、embedding 语义重复检测、用户反馈闭环（错题标记→触发重写）

### Q23：如果重新做，你会怎么改？

**要点**：
- Agent 自检：每个 Agent 产出后先自检（如 cleaner 自检分类是否合理），减少主理人核验压力
- 基准知识库：构建大数据事实库（如 Kafka 参数正确值表），答案比对不一致自动触发重写
- Embedding 重复检测：Jaccard 只抓字面重复，加一层 embedding 抓语义重复
- 流水线可视化：实时展示各阶段进度和评测分数

### Q24：这个项目你学到了什么？

**要点**（真诚反思，不吹）：
- **方法论**：多 Agent 协作的关键不是"多"，而是"职责清晰 + 质检兜底"——没有 Harness 的多 Agent 就是失控的乱炖
- **工程思维**：可量化 > 可运行 > 可复现，"我觉得好"不算数，得有数字证明
- **踩坑经验**：GitHub Secret Scanning、Contents API、网络阻断排查——这些是真实工程问题，不是教科书内容
- **局限认知**：当前没有基准库交叉校验、Jaccard 有局限、Agent 自检缺失——知道哪里不够好，比假装完美重要

## 回答策略提醒

| 问题类型 | 策略 |
|---------|------|
| 是什么 | 先说结论，再展开 1-2 句 |
| 为什么 | 先说痛点/问题，再说方案 |
| 怎么做 | 先说整体思路，再说关键细节 |
| 量化数据 | 直接报数字，不修饰 |
| 局限/反思 | 真诚承认，给改进方向，不找借口 |

> **铁律**：被问到不会的，说"这个我目前没做，但如果要做的话会这样设计……"——展示思考能力比假装全懂更有价值。
