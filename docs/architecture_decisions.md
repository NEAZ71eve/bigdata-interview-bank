## ADR-001：采用多 Agent 流水线而非单模型生成

### Context

需要从原始面经生产高质量题库，涉及采集、清洗、出题、阅卷、质检五种不同性质的工作。

### Options Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| 单模型一次生成 | 简单、快 | 上下文窗口放不下 100 道题+答案、角色混乱、质量不可控 |
| 多模型串行调用 | 比单模型好 | 仍是线性流程，无并行、无角色隔离 |
| **多 Agent 协作流水线** | 职责拆分、可并行、可独立迭代 | 协调成本高、需要消息中转机制 |

### Decision

采用多 Agent 协作流水线，5 个专职 Agent 串行联动 + 阶段内并行。

### Consequences

- **正面**：每个 Agent prompt 聚焦一件事，质量天花板高；Phase 4 两个阅卷 Agent 并行节省时间；Agent 定义可独立迭代不影响其他
- **负面**：主理人协调成本增加；Agent 间不能直接通信，信息传递多一跳；需要设计阶段门禁机制
- **权衡**：协调成本用主理人中转模式消化，换取可控性和可追溯性

## ADR-002：Agent 间禁止直接通信，所有信息经主理人中转

### Context

5 个 Agent 需要传递产出文件和上下文信息。

### Options Considered

| 方案 | 通信复杂度 | 上下文污染 | 可追溯性 |
|------|-----------|-----------|---------|
| 全互联（任意两两通信） | O(N²) | 严重 | 差 |
| 总线（发布订阅） | O(N) | 中等 | 中等 |
| **主理人中转** | O(N) | 无 | **好** |

### Decision

所有跨 Agent 信息流经主理人中转，Agent 之间不直接 SendMessage。

### Consequences

- **正面**：主理人掌握全局视图，可做质量核验和阶段门禁；上下文零污染（每个 Agent 只看到主理人发给它的内容）；消息链路清晰可追溯
- **负面**：每次传递多一跳（Agent→主理人→下一Agent），延迟略增
- **权衡**：延迟增加可忽略（秒级），换取可控性大幅提升

## ADR-003：Harness 评测用 Jaccard 相似度而非 embedding

### Context

需要检测题库中是否有重复题目。

### Options Considered

| 方案 | 依赖 | 速度 | 能抓语义重复 | 能抓字面重复 |
|------|------|------|------------|------------|
| **Jaccard 字符 bigram** | 零 | 秒级（6216 对） | 不能 | 能 |
| TF-IDF + 余弦 | 零 | 秒级 | 部分 | 能 |
| Embedding 余弦 | 需模型 | 分钟级 | **能** | 能 |

### Decision

选用 Jaccard 字符 bigram 相似度，阈值 0.5。

### Consequences

- **正面**：零依赖、可解释、速度快、可复现；实测 112 道题 0 重复，当前够用
- **负面**：语义相似但字面不同会漏检（如"Kafka 怎么保证不丢消息" vs "Kafka 消息可靠性机制"）
- **改进方向**：题量增大后可加 embedding 层做语义重复检测，但需要引入模型依赖，当前 Jaccard 作为 baseline 足够

## ADR-004：答案准确性用参数密度而非语义正确性

### Context

需要量化答案质量，但"答案对不对"是主观判断。

### Options Considered

| 方案 | 可量化 | 需基准库 | 能评正确性 | 能评专业性 |
|------|--------|---------|-----------|-----------|
| 人工评分 | 不能 | 不需要 | 能 | 能 |
| LLM 评分 | 能 | 不需要 | 部分 | 部分 |
| **真实参数密度** | **能** | 不需要 | 不能（代理指标） | **能** |

### Decision

用正则匹配真实配置参数/API 的命中密度（次/题）作为答案专业性的代理指标，阈值 2/题。

### Consequences

- **正面**：零依赖、可复现、客观可量化；参数多 ≠ 答案对，但参数少 ≈ 答案不专业，作为下限指标有效
- **负面**：不能检测"参数对但逻辑错"的情况；可能鼓励堆参数而非讲清楚原理
- **权衡**：这是 proxy metric，不完美但在没有基准知识库的情况下是最实际的方案；实测密度 7.4/题远超阈值，说明答案确实参数丰富

## ADR-005：难度均衡度容差 ±7%

### Context

需要判断 L1-L4 占比是否合理。

### Options Considered

| 容差 | 误报率 | 漏报率 | 适用场景 |
|------|--------|--------|---------|
| ±3% | 高 | 低 | 严格品控 |
| ±5% | 中 | 中 | 平衡 |
| **±7%** | 低 | 中低 | **实用导向** |
| ±10% | 极低 | 高 | 过于宽松 |

### Decision

选用 ±7% 容差。

### Consequences

- **正面**：首次 L1 偏差 -12% 正确触发 FAIL，说明阈值有效；不会因微小偏差频繁误报
- **负面**：±7% 意味着 L1 8%-22% 都算合格，区间较宽
- **权衡**：阈值太严会频繁阻断流水线，太松失去意义；±7% 是经验值，可按场景调（刷题产品可放宽 L1，面试冲刺可收紧 L4）

## ADR-006：用 GitHub Contents API 逐文件推送而非 git push

### Context

需要将代码推送到 GitHub 远程仓库，但 `github.com` SSL 握手被重置（IP 20.205.243.166 连接失败），`git push` 不可用。而 `api.github.com` 可达（HTTP 200）。

### Options Considered

| 方案 | 可行性 | 原子性 | 大文件支持 |
|------|--------|--------|-----------|
| 配置 HTTP 代理 | 取决于代理可用性 | 好 | 好 |
| 换网络环境 | 不确定 | 好 | 好 |
| **Contents API 逐文件** | **已验证可行** | 差（每文件一 commit） | <1MB |
| Git Database API（blob+tree+commit） | 可行但复杂 | 好 | 好 |

### Decision

用 Contents API 逐文件推送，每个文件先 GET 获取 sha（已存在则更新）、再 PUT 上传内容。

### Consequences

- **正面**：绕过 github.com 网络阻断，11 个文件全部推送成功；纯 Python 标准库实现，零依赖
- **负面**：每文件一次 commit，不是原子提交；更新文件需先 GET sha 再 PUT，多一次往返；大文件需 base64 编码放 JSON body
- **改进方向**：如果网络恢复，切回 git push；或用 Git Database API（blob+tree+commit）实现原子提交

## ADR-007：Harness 评测引擎用纯 Python 标准库而非引入框架

### Context

需要实现四维评测引擎，涉及文本相似度、正则匹配、统计计算、报告生成。

### Options Considered

| 方案 | 依赖 | 可移植性 | 功能 |
|------|------|---------|------|
| pytest + assert | pytest | 中 | 仅断言，无报告 |
| **纯 Python 标准库** | **零** | **极好** | **全功能** |
| 引入 NLP 库 | numpy/sklearn | 差 | 更强但过重 |

### Decision

纯 Python 标准库实现（re / json / os / urllib / collections），零第三方依赖。

### Consequences

- **正面**：任何有 Python 3.8+ 的环境直接能跑，无需 pip install；可移植性极强；代码可读性高
- **负面**：Jaccard 实现较原始（手写 bigram）；没有 pytest 的 fixture 参数化
- **权衡**：评测引擎的核心价值是"可量化 + 可复现"，零依赖比功能丰富更重要
